# Architecture

## Overview

Single executable desktop application built with **Tauri 2**:
- **Frontend**: React 19 with TypeScript
- **Backend**: Rust with SQLite (sqlx)
- **IPC**: Specta-generated type bindings (`src/bindings.ts`)

---

## Backend Architecture (`src-tauri/src/`)

The Rust backend follows **Domain-Driven Design** with bounded contexts.

### Structure

```
src-tauri/src/
├── lib.rs                      # App setup, dependency wiring, Tauri state management
├── core/
│   └── specta_builder.rs       # Tauri command registration + Specta type export
├── context/                    # Bounded contexts (self-contained domain modules)
│   ├── bank/                   # Bank accounts & transfers
│   │   ├── domain/             # Entities: BankAccount, BankTransfer
│   │   ├── repository/         # Traits + SQLite impls
│   │   ├── service.rs          # BankTransferService, BankAccountService
│   │   └── api.rs              # Tauri command handlers
│   ├── fund/                   # Fund payment groups & lines
│   │   ├── domain.rs           # FundPaymentGroup, FundPaymentLine
│   │   ├── repository.rs       # Trait + SQLite impl
│   │   └── service.rs          # FundPaymentService
│   ├── patient/                # Patients
│   ├── procedure/              # Procedures (acts)
│   └── excel_import/           # Excel parsing (amount mapping)
├── use_cases/                  # Cross-context orchestrators
│   ├── bank_manual_match/      # Manual bank reconciliation (R6–R20)
│   │   ├── orchestrator.rs     # BankManualMatchOrchestrator
│   │   └── api.rs              # 12 Tauri commands
│   ├── bank_statement_reconciliation/  # Automatic PDF reconciliation
│   ├── excel_import/           # Excel import orchestration
│   ├── fund_payment_reconciliation/    # Fund payment matching
│   └── procedure_orchestration/       # Procedure batch operations
└── migrations/                 # SQLite schema migrations
```

### Key Patterns

**Factory Methods** — All domain entities use:
- `new()` — generate ID + validate (create flow)
- `with_id()` — provided ID + validate (Tauri command flow)
- `restore()` — no validation (database read)

**Repository Traits** — Services depend on `Arc<dyn XxxRepository>`, never concrete types.

**Events** — Bounded contexts publish `{Domain}Updated` events after state changes. Use-cases do NOT publish events directly.

**Error handling** — Services return `anyhow::Result<T>`. Tauri handlers return `Result<T, String>`.

## Frontend Architecture (`src/`)

### Structure

```
src/
├── bindings.ts                 # Auto-generated by Specta — DO NOT EDIT
├── features/                   # Feature modules (DDD-organized)
│   ├── bank-transfer/          # Bank transfer management
│   │   ├── manual_match/       # Manual match gateway + selection panels
│   │   ├── add_bank_transfer_form/
│   │   ├── bank_transfer_list/
│   │   ├── edit_bank_transfer_modal/
│   │   └── shared/
│   ├── bank-account/
│   ├── bank-statement/         # PDF auto-reconciliation
│   ├── fund-payment/           # Fund payment groups
│   ├── fund-payment-match/     # Fund payment auto-match
│   ├── patient/
│   ├── procedure/
│   └── excel-import/
├── core/                       # App-wide infrastructure
│   ├── snackbar/               # Toast notifications
│   └── events/                 # Window event bus
├── lib/
│   ├── appStore.ts             # Global Zustand store (patients, funds, bankAccounts…)
│   └── logger.ts               # Structured logging
├── i18n/                       # Translations (fr / en)
└── ui/components/              # Shared UI components (Button, DateField, SelectField…)
```

### Feature Module Pattern

Each feature folder follows this structure:

```
features/{domain}/
├── gateway.ts                  # Service layer: wraps Tauri commands, returns ServiceResult<T>
├── store.ts                    # Feature-scoped Zustand store (if needed)
├── {feature}_list/             # List component + hook
├── add_{feature}_form/         # Add form component + hook
├── edit_{feature}_modal/       # Edit modal component
└── shared/                     # Validators, presenters, types
```

### Data Flow

```
Component → hook (business logic) → gateway (Tauri call) → Rust service → SQLite
                                                     ↓
                        Backend publishes {Domain}Updated event
                                                     ↓
                        useAppInit dispatches window event
                                                     ↓
                        useBankTransferOperations refetches → store → UI re-renders
```

### Key Patterns

**Gateway** — All `commands.*` calls are wrapped in a `gateway.ts` with `ServiceResult<T>` return. Components never call `commands.*` directly.

**Tauri bindings** — Parameters must match `bindings.ts` signatures exactly (positional, not wrapped).

**Global store** — `useAppStore` holds shared data (patients, funds, bankAccounts, fundPaymentGroups). Feature stores hold feature-scoped UI state.

## Bounded Contexts & Use Cases

| Module | Location | Responsibility |
|--------|----------|----------------|
| Bank transfers | `context/bank/` | CRUD for bank accounts and transfers |
| Fund payments | `context/fund/` | Fund payment groups and lines |
| Patients | `context/patient/` | Patient records |
| Procedures | `context/procedure/` | Acts (procedures) |
| **Manual match** | `use_cases/bank_manual_match/` | Manual bank reconciliation: link transfers to fund groups (R6–R12) or procedures (R14–R20) |
| Auto reconciliation | `use_cases/bank_statement_reconciliation/` | PDF bank statement parsing and auto-matching |
| Fund payment match | `use_cases/fund_payment_reconciliation/` | Match procedures to fund payment groups |
| Excel import | `use_cases/excel_import/` | Bulk procedure import from Excel |
| Procedure orchestration | `use_cases/procedure_orchestration/` | Batch procedure creation |

## Database

SQLite via `sqlx` with compile-time query verification.

- Schema managed via migrations in `src-tauri/migrations/`
- After schema changes: run `just clean-db` then `cargo sqlx prepare` to update `.sqlx/` query cache
- Never add `BEGIN`/`COMMIT` in migration files (sqlx wraps each in a transaction)
